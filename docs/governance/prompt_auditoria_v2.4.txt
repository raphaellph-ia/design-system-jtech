🧱 PREMISSAS ARQUITETURAIS ATUAIS (OBRIGATÓRIAS)

1️⃣ Compact Controls (OFICIAL)
Considere como CONFORME:
Uso de tokens:
- --dss-compact-control-height-xs
- --dss-compact-control-height-sm
- --dss-compact-control-height-md
- --dss-compact-control-height-lg
- Altura visual menor que touch target
- Componentes densos (Chip, Badge, etc.)
❌ NÃO sugerir tokens específicos de componente (--dss-chip-height-*, --dss-badge-height-*)

2️⃣ Touch Target vs Altura Visual
É obrigatório reconhecer como válido:
Touch target ≥ tokens:
- --dss-touch-target-xs
- --dss-touch-target-sm
- --dss-touch-target-md
- --dss-touch-target-lg
Expansão de área clicável via:
- ::before
- wrapper interno
- pseudo-elementos não visuais
⚠️ Só marque erro se:
- O touch target for perdido em runtime
- A solução violar WCAG 2.5.5

3️⃣ Convenção de Pseudo-elementos (OFICIAL)
Considere como padrão válido:
- ::before → reservado para touch target
- ::after → efeitos visuais (hover, overlay, selected, ripple, etc.)
❌ Só marque não-conformidade se:
- Um pseudo-elemento sobrescrever o outro funcionalmente

4️⃣ Regime de Exceções Documentadas
Valores hardcoded são PERMITIDOS quando:
- Operam sobre contexto dinâmico: filter: brightness(), saturate()
- São ignorados em forced-colors / high-contrast
- Estão documentados no código ou README
- Estão respaldados no DSS_IMPLEMENTATION_GUIDE.md
❌ NÃO sugerir tokenização nesses casos.

5️⃣ Modelo Dual Golden (OFICIAL)
O DSS adota modelo dual:
- Golden Reference → Referência normativa global oficialmente declarada (ex.: DssChip, DssBadge)
- Golden Context → Baseline específico usado para auditar um componente
- Golden Sample → Referência de documentação (Template 13.1), NÃO arquitetural
Regras para auditoria:
- Auditoria deve usar Golden Context como baseline principal
- Golden Reference deve ser usado como verificação transversal
- Golden Sample NÃO deve ser usado como baseline arquitetural
- Não reinterpretar Golden Reference como Golden Context
- Não alterar nem questionar Golden References oficialmente declaradas

🔍 O QUE AUDITAR (CHECKLIST)

A. Tokens
- Tokens inexistentes ou inválidos ❌
- Uso correto de tokens genéricos ✅
- Nenhum token específico de componente novo ❌

B. Arquitetura
🔒 Gate Estrutural DSS (CLAUDE.md — fonte de verdade)
Deve validar obrigatoriamente:
- Presença das 4 camadas obrigatórias (L1 / L2 / L3 / L4)
- Orquestrador SCSS (.module.scss) importa L2 → L3 → L4 na ordem correta
- Entry Point Wrapper obrigatório:
  - Existe DssNomeComponente.vue na raiz do componente
  - É re-export puro da Layer 1
  - Sem template próprio, sem <style>, sem lógica adicional
  - Aponta corretamente para ./1-structure/DssNomeComponente.ts.vue
- index.js exporta o wrapper como entry point principal
❗ Ausência do Entry Point Wrapper obrigatório deve ser classificada como NÃO-CONFORMIDADE BLOQUEANTE.

🔒 Gate de Composição (Fase 2 - NOVO v2.4)
- Zero HTML Nativo Substituível: O componente NÃO PODE usar tags HTML nativas (`<button>`, `<a>`, `<input>`, `<img>`) se existir um componente DSS equivalente (`<DssButton>`, `<DssItem>`, `<DssInput>`, `<DssAvatar>`).
- Zero Quebra de Encapsulamento: O componente pai NÃO PODE usar seletores CSS (como `:deep()`, `::v-deep` ou descendência direta) para alterar a aparência de um subcomponente DSS filho. Toda alteração visual no filho deve ser feita via props do próprio filho.
- Importação Correta: Componentes DSS internos DEVEM ser importados via seus wrappers na raiz (ex: `../DssButton/DssButton.vue`), NUNCA via `1-structure`.

🔒 Gate de Responsabilidade (Fase 2 - NOVO v2.4)
- Delegação de Estados: Componentes container (ex: DssBtnGroup, DssList) NÃO PODEM capturar estados interativos (hover, focus, active) que semanticamente pertencem aos filhos. Verifique se o SCSS do componente pai define `:hover`, `:focus` ou `:active` em elementos que são renderizados por componentes DSS filhos.
- Sem Lógica de Negócio: O componente deve resolver exclusivamente um padrão de UI/UX. Verifique se há lógica condicional no `<script>` que dependa de regras de negócio específicas de um produto (Hub, Water, Waste) em vez de props genéricas.
- Limites Documentados: A documentação (`.md`) DEVE declarar explicitamente o que o componente faz e o que ele delega aos filhos. Ausência dessa declaração é Gap documentável.
❌ Marque como NÃO-CONFORMIDADE BLOQUEANTE se o componente capturar estados de filhos via CSS ou misturar lógica de negócio no template.

C. Estados
- hover, focus, active, disabled
- loading: Implementado ou Justificado explicitamente como ausente

D. Acessibilidade
- WCAG 2.1 AA (mínimo)
- Touch target válido
- Navegação por teclado
- ARIA coerente
- prefers-reduced-motion, prefers-contrast, forced-colors

E. Documentação
- README consistente com código
- API documentada ≠ API real ❌
- Exceções claramente listadas
- Nenhuma ambiguidade interpretativa

F. Validação de Pré-prompt (Fase 2 - NOVO v2.4)
Verifique se existe o arquivo `docs/governance/pre-prompts/pre_prompt_dss_[nome].md` e se ele cobriu os 5 eixos obrigatórios. Cada eixo ausente no pré-prompt que gerou uma não-conformidade no componente deve ser registrado como Gap para correção retroativa do pré-prompt.
- Classificação e Contexto: Fase, Nível, Golden Context e justificativa de Fase 2 estão declarados?
- Grande Risco Arquitetural: O "calcanhar de Aquiles" do componente foi identificado com anti-patterns e padrões corretos?
- Mapeamento de API: Props expostas, bloqueadas e justificativas estão listadas?
- Governança de Tokens: Os tokens específicos para bordas, espaçamentos e divisóres foram mapeados?
- Acessibilidade e Estados: Role ARIA, decisão de touch target e delegação de estados estão definidos?
⚠️ Ausência do pré-prompt ou cobertura incompleta dos 5 eixos é Gap documentável (não bloqueante para o componente, mas bloqueante para o próximo componente da mesma família).
